# BYOCA (Bring Your Own CA) — Current State

**Audience:** Org operators, security teams, enterprise integration teams.
**Status:** Accurate as of 2026-05-18.

---

## What BYOCA means

BYOCA ("Bring Your Own CA") refers to an enrollment model where an organization
uses its own Certificate Authority (CA) to sign agent certificates, rather than
relying on a Cullis-generated CA. In the true definition, the organization's
external PKI (HashiCorp Vault, AWS ACM Private CA, Microsoft ADCS, step-ca, or
any standard x509 CA) acts as the trust anchor that Cullis validates agent
credentials against.

There are two subtly different things that Cullis documentation has called BYOCA:

1. **Court-level BYOCA (fully implemented):** An organization registers its own
   CA certificate with the Court (broker). When an agent presents a cert at
   login, the Court walks the chain to that registered CA. The private key of
   this CA never leaves the organization's infrastructure.

2. **Mastio-level BYOCA (partially implemented, with a key gap):** The Mastio
   has a BYOCA enrollment endpoint, but it only accepts certs signed by the
   Mastio's own Org CA, not by an arbitrary external PKI.

## What exists in the code

### 1. Court-level org CA trust (fully implemented)

The Court stores one CA certificate per organization (`orgs.ca_certificate`
column). When an agent authenticates via `POST /v1/auth/token`, the verifier
walks the presented x5c chain to the registered Org CA:

- `app/auth/x509_verifier.py:374-407` — loads `org.ca_certificate` from the DB,
  verifies it is a true CA (BasicConstraints CA=true), then calls `_walk_chain`
  to validate the agent cert chain against it.

The CA can be externally generated by the organization itself, as documented in
`enterprise-kit/BYOCA.md`. The workflow is:

1. Organization generates its own CA (any x509 CA).
2. Organization registers with the Court, submitting the CA cert
   (`/onboarding/join` or the dashboard).
3. Organization issues agent certs signed by that CA.
4. The Court verifies agent certs against the registered CA at login time.

This **is** real BYOCA at the Court level. The organization's CA private key
never touches Cullis.

Relevant files:
- `enterprise-kit/BYOCA.md` — step-by-step integration guide
- `app/auth/x509_verifier.py:240-569` — full verification pipeline
- `app/auth/router.py:242-268` — thumbprint pinning for BYOCA mode (classic,
  non-SPIFFE path)
- `app/dashboard/router.py:2363-2438` — dashboard cert upload endpoint for
  agents ("Upload an externally signed agent certificate — BYOCA production
  flow")

### 2. Mastio BYOCA enrollment endpoint (implemented, with scope limitation)

`POST /v1/admin/agents/enroll/byoca` (`mcp_proxy/admin/enroll.py:217-349`)
enrolls an agent by accepting a caller-supplied cert+key pair. The key
validation step:

```python
# mcp_proxy/admin/enroll.py:273
if not _verify_signed_by_ca(cert, org_ca):
    raise HTTPException(
        status_code=status.HTTP_400_BAD_REQUEST,
        detail="cert_pem is not signed by this Mastio's Org CA",
    )
```

The `org_ca` here is the Mastio's own Org CA loaded from its local Vault/disk
(`getattr(mgr, "_org_ca_cert", None)` at line 266). This means the submitted
cert must be signed by the Mastio's Org CA, not by an external CA the
organization might already operate.

The enrollment method field is stored as `'byoca'` in the `internal_agents`
table (`mcp_proxy/admin/enroll.py:311`).

The `enrollment_method` column was introduced by
`mcp_proxy/alembic/versions/0016_enrollment_metadata.py`, with allowed values
`connector | byoca | spiffe`.

### 3. Legacy Court direct-login path (deprecated)

`POST /v1/auth/token` (`app/auth/router.py:53-332`) is the original "BYOCA
direct-login" path where agents with an externally-issued cert authenticate
directly to the Court. This endpoint is now marked deprecated (ADR-011 Phase 3):
it emits `Deprecation: true` and `Sunset` response headers on every 200, and
logs `auth_mode: "byoca"` in the audit stream. The migration target is
`/v1/admin/agents/enroll/byoca` on the Mastio.

From `app/auth/router.py:32`:
> "legacy SPIFFE/BYOCA direct-login path. Under the unified enrollment model
> agents authenticate to their Mastio with API-key + DPoP, and never hit this
> endpoint."

### 4. SDK enrollment helpers

The Python SDK (`cullis_sdk/`) exposes `enroll_via_byoca(...)` as a factory
method. From `CHANGELOG.md:449`, `ca_chain_path` was threaded through
`enroll_via_byoca`, `enroll_via_spiffe`, and `from_api_key_file` in v0.3.7.

### 5. Note on user principal signing

`mcp_proxy/principals_csr.py:15-17` explicitly notes:

> "Future BYOCA enterprise deployments will need a different signer that
> delegates to the org's KMS — that's a follow-up beyond v0.1."

This confirms the limitation is known for user principal cert signing as well.

## The gap

The Mastio BYOCA enrollment endpoint (`POST /v1/admin/agents/enroll/byoca`)
validates that the submitted cert is signed by the **Mastio's own Org CA**.
It does not accept certs signed by an organization's pre-existing external PKI.

This creates a discrepancy for organizations that:
- Already operate a corporate PKI (Microsoft ADCS, AWS ACM-PCA, Venafi, etc.)
- Want agent certs issued by that existing PKI to authenticate to the Mastio
- Cannot or do not want to make the Mastio's Org CA their primary agent signing CA

In the current model, such organizations have two options:
1. Accept the Mastio-generated Org CA as the signing root for Cullis agents
   (separate from their corporate PKI).
2. Replace the Mastio's Org CA with their external CA by configuring the
   Mastio to load the external CA's key at startup. This works but requires
   the external CA's private key to be accessible to the Mastio, which may
   conflict with HSM or CA policy.

There is no path today to configure the Mastio to accept certs signed by an
additional trusted external CA while keeping its own Org CA operational.

---

## What we currently offer instead

**For orgs comfortable with a Cullis-managed Org CA:**

The Mastio auto-generates an Org CA at initialization. Agents are enrolled via:

- `POST /v1/enrollment/*` — Connector device-code flow, where the Mastio signs
  the agent cert with its Org CA.
- `POST /v1/admin/agents/enroll/byoca` — operator-supplied cert+key, verified
  against the Mastio's Org CA.
- `POST /v1/admin/agents/enroll/spiffe` — SVID enrollment, verified against
  a SPIRE trust bundle.

The Org CA private key is protected by the Mastio's KMS backend (local
filesystem or HashiCorp Vault, per ADR-031).

**For orgs with their own external CA (Court-registered path):**

The Court-level BYOCA is fully operational. Organizations register their own CA
cert with the Court, issue agent certs with their own CA, and authenticate
directly. See `enterprise-kit/BYOCA.md` for the step-by-step guide. Note that
this path uses the legacy `POST /v1/auth/token` endpoint, which is deprecated
(ADR-011) and will return `410 Gone` at sunset.

The current migration path from Court-level BYOCA to the Mastio enrollment model
requires re-issuing agent certs under the Mastio's Org CA, which breaks the
"your CA signs agent certs" promise for organizations that need to preserve
their external PKI chain.

## Summary of implementation status

| Capability | Implemented | Notes |
|---|---|---|
| Court registers external org CA as trust anchor | Yes | `orgs.ca_certificate`, verified at login |
| Agent certs signed by external CA authenticate to Court | Yes | Legacy path, deprecated (ADR-011) |
| Dashboard cert upload for agents (external cert) | Yes | `app/dashboard/router.py:2363`, verifies against org CA |
| Mastio enrollment with pre-signed cert | Yes | `mcp_proxy/admin/enroll.py`, cert must be signed by Mastio's Org CA |
| Mastio accepts certs from additional trusted external CA | No | No `trusted_external_cas` table or additive CA trust |
| nginx dynamic `ssl_client_certificate` for external CA | No | Static Mastio Org CA only |
| User principal certs signed by external KMS CA | No | Noted as future work in `principals_csr.py:15` |
